# 文档先行工作流

> 定义"先文档、后实现、再验收"的开发流程，适用于 L1 及以上变更。

## 一、基本原则

1. **文档先于代码**：任何功能开发、重构、缺陷修复，先补文档再改代码
2. **小改动也要记录**：即使是 L0 纯文案调整，也要补变更记录和验收点
3. **文档与实现同步**：代码完成后必须回写文档，保持一致
4. **无文档不算完成**：没有对应文档更新的实现变更，视为未完成

## 二、变更分级

### L0：纯文案/样式微调

- **适用**：文案修改、样式调整、低风险显示修正
- **要求**：直接改，但要补简短变更记录（改了什么 + 验收点）
- **输出**：变更记录

### L1：单页面逻辑或交互变更

- **适用**：单页面内的逻辑、交互、组件行为调整
- **要求**：先写轻量功能说明，再实施
- **输出**：功能规格简述 + 验收清单

### L2：数据模型/配置/架构/跨页面变更

- **适用**：数据结构变化、配置项新增或语义变更、架构调整、共享工具修改、云函数契约变化
- **要求**：先写完整功能规格（含决策记录和兼容/迁移说明），再实施
- **输出**：完整规格文档 + 验收清单 + 必要时的 ADR

## 三、工作流程

```
需求发现/任务接收
    ↓
判定变更级别（L0/L1/L2）
    ↓
┌─────────────────────────────────────┐
│ L0: 补变更记录                          │
│   → 改代码                             │
│   → 自测验证                           │
│   → 记录完成                           │
├─────────────────────────────────────┤
│ L1: 写功能规格简述                       │
│   → 评审规格（可选）                      │
│   → 改代码                             │
│   → 对照验收清单验证                      │
│   → 完成                              │
├─────────────────────────────────────┤
│ L2: 写完整规格文档 + ADR（如需要）         │
│   → 评审规格（建议）                      │
│   → 改代码                             │
│   → 对照验收清单验证                      │
│   → 若有兼容/迁移问题，写 ADR             │
│   → 完成                               │
└─────────────────────────────────────┘
```

## 四、验收清单要求

每次变更后必须验证以下主流程未被破坏：

- [ ] 创建房间
- [ ] 加入房间
- [ ] 记交易（普通转账）
- [ ] 撤销交易
- [ ] 茶水费收取
- [ ] 台面操作
- [ ] 结算
- [ ] 历史记录
- [ ] 个人统计
- [ ] 分析页

若无法运行验证（如无微信开发者工具），须在交付中明确：
- 未验证项
- 存在风险

## 五、交付格式要求

每次交付必须包含：

```
## 改了什么
（简洁描述）

## 为什么改
（动机，可引用需求来源）

## 影响范围
（涉及的文件、页面、接口）

## 验证结果
（对照验收清单，未验证项需说明）

## 剩余风险
（如有）
```

## 六、模板位置

详细模板见 `docs/templates/`：
- `feature-spec.md` — 功能规格模板（L1/L2）
- `refactor-spec.md` — 重构说明模板
- `bugfix-spec.md` — 缺陷修复说明模板
- `adr.md` — 决策记录模板
- `checklist.md` — 手工验收清单模板
